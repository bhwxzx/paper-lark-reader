# 追问与启发讨论

## 触发条件

用户围绕已读取的论文正文、已生成笔记或当前对话中的阅读结果提问，要求解释知识点、讨论方法迁移或研究启发时使用。

首次给出 PDF 并要求阅读、精读、总结或分析时，不使用本流程，必须转入 `paper-read.md`。

普通讨论不得自动进入 `note-revise.md`。只有用户明确要求“写回笔记”“更新笔记”“同步这次讨论”时，才转入修订流程。

## 前置

优先读取论文正文、已生成笔记或当前对话上下文。需要理解笔记结构时读取 `references/note-writing-guide.md`。

若用户询问“对我的研究有什么启发”，可读取 `<SKILL_ROOT>/config/defaults.json`。`<SKILL_ROOT>` 是包含 `SKILL.md` 的 `paper-lark-reader/` 目录，不是当前项目目录：

- 若存在可访问的 `space_id`、`app_token` 和 `table_id`，只读获取 `研究概述` 和已有论文索引字段后回答。
- 若配置缺失或权限不足，只基于论文内容和本轮信息回答，并说明原因。

## 回答策略

1. 优先基于论文正文和已生成笔记回答。
2. 明确区分论文作者结论、阅读者推断和你的建议。
3. 若引用画像，只说明画像如何影响启发，不把画像内容写成论文作者观点。
4. 不自动修改本地笔记、云端文档、Base 记录或研究画像。

## 候选补充记录

每次有可沉淀内容时，在对话中保留以下结构，供后续修订使用：

| 字段 | 内容 |
|---|---|
| 用户问题 | 本轮问题 |
| 回答要点 | 可沉淀的结论 |
| 建议章节 | 元数据 / 背景与问题 / 方法 / 数据 / 实验 / 评价与启发 / 补充信息 |
| 是否属于启发 | 是 / 否 |

候选补充不等于更新笔记；除非用户明确要求写回，否则只保留在当前对话上下文中。

## 反例

- 不要把每次追问都自动追加到笔记。
- 不要在讨论时更新 `研究概述`。
- 不要把读者推断说成论文作者结论。
- 不要在首次给出 PDF 并要求阅读时走本讨论流程；必须走 `paper-read.md`。
