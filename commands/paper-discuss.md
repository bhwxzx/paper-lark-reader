# 追问与启发讨论

## 触发条件

用户围绕已读取的论文正文、已生成笔记、全文翻译或当前对话中的阅读结果提问，要求解释知识点、讨论方法迁移或研究启发时使用。

首次给出 PDF 并要求阅读、精读、总结或分析时，不使用本流程，必须转入 `paper-read.md`。

普通讨论不得自动进入 `note-revise.md`。只有用户明确要求“写回笔记”“更新笔记”“同步这次讨论”时，才转入修订流程。

## 前置动作（资料检索策略）

为了准确回答用户问题，请优先查阅以下本地文件（按需读取）：
1. **论文笔记**：位于目标论文所在目录的 `notes/*.paper-note.md`。
2. **全文翻译**：位于目标论文所在目录的 `translation/*_translated.md`。
3. **写作规范**：需要理解笔记结构时读取 `references/note-writing-guide.md`。

**关于“研究启发”的特殊处理**：
若用户询问“对我的研究有什么启发”：
- **首选**：直接读取本地工作区根目录下的 `研究概述.md`。
- **备选**：若本地无此文件，再读取 `<SKILL_ROOT>/config/defaults.json`，若存在可访问的 Token 配置，则使用 `npx lark-cli docs +fetch` 只读获取云端 `研究概述`。
- 若两者皆无，则只基于论文内容回答，并说明缺少研究画像上下文。

## 回答策略

1. **基于事实**：优先基于论文正文、笔记和翻译文件回答，不要凭空捏造论文中不存在的算法细节。
2. **学术严谨**：在聊天窗口中解释控制算法、状态空间或奖励函数时，**必须同样使用标准的飞书兼容 LaTeX 语法（单美元符号 `$math$`，不带空格）**。
3. **观点隔离**：明确区分“论文作者结论”、“阅读者（AI）推断”和“基于用户画像的建议”。
4. **只读约束**：纯讨论状态下，**绝对禁止**自动修改本地笔记、云端文档、Base 记录或研究画像。

## 候选补充记录

每次回答中，如果有非常有价值的、可沉淀到笔记里的内容（如深度的推导解释、新的启发），请在回答的末尾，使用以下标准的 Markdown 表格向用户展示“候选补充记录”，供后续修订使用：

| 字段 | 内容 |
| :--- | :--- |
| **用户问题** | 概括本轮追问的核心问题 |
| **回答要点** | 提炼可沉淀到笔记中的高价值结论 |
| **建议章节** | 指明应补充到哪个章节（元数据 / 核心概括 / 背景与问题 / 方法 / 数据与环境 / 实验 / 评价与启发 / 补充信息） |
| **是否属于启发** | 是 / 否 |

*提示语：展示表格后，请附上一句话：“候选补充不等于更新笔记；除非您明确要求写回/更新笔记，否则本记录仅保留在当前对话中。”*

## 反例（禁止行为）

- 绝对不要把每次追问都自动追加或改写到 `.paper-note.md` 中。
- 绝对不要在讨论论文时，擅自更新修改 `研究概述.md`。
- 不要把读者的推断或常识说成是论文作者的结论。
- 不要在首次给出 PDF 并要求阅读时走本讨论流程；必须走 `paper-read.md`。
